### 1. Keep It Simple, Stupid (KISS)

The KISS principle serves as a reminder to prioritize simplicity in your coding approach. Rather than introducing unnecessary complexity or attempting to showcase your skills by composing verbose code blocks, focus on clarity.
If a problem can be solved gracefully in a single line, embrace that straightforward solution.

Example without following the KISS principle:

```ts title="example.ts"
const calculateAverage = (numbers: number[]): number => {
  let sum = 0;

  for (let i = 0; i < numbers.length; i++) {
    sum += numbers[i];
  }

  const average = sum / numbers.length;
  return parseFloat(average.toFixed(2));
};

const numbers = [1, 2, 3, 4, 5];
const average = calculateAverage(numbers);

console.log(average);
```

In the snippet above, the KISS principle is ignored, resulting in needless complexity.
Using a manual loop and the **toFixed** method to restrict decimal formatting makes the function unnecessarily bulky.
Such overengineering not only increases the risk of introducing bugs but also makes the codebase harder to maintain and comprehend.

Let's simplify it:

An improved approach leveraging the KISS principle:

```ts title="example.ts"
const calculateAverage = (numbers: number[]): number => {
  const sum = numbers.reduce((acc, curr) => acc + curr, 0);
  return sum / numbers.length;
};

const numbers = [1, 2, 3, 4, 5];
const average = calculateAverage(numbers);

console.log(average);
```

By applying the KISS approach, we compute the numeric average using a far more elegant and concise method.
We utilize the **reduce** function to calculate the total sum, followed by a simple division by the array's length to find the average.
The resulting code is much more legible, direct, and accomplishes the goal without any bloated logic.

### 2. Don't Repeat Yourself (DRY)

The core philosophy behind this principle is simple: never duplicate logic.
Suppose you need the same specific functionality across various parts of your application.
Rather than blindly copying and pasting the identical code block everywhere, the DRY heuristic advises you to abstract that logic into reusable functions, classes, components, or modules.

<blockquote>
  Every piece of knowledge must have a single, unambiguous, authoritative
  representation within a system.
</blockquote>

Example without following the DRY principle:

```ts title="example.ts"
const number1 = 1;
const number2 = 2;
const sum1 = number1 + number2;
console.log(sum1);

// ...

const number3 = 3;
const number4 = 4;
const sum2 = number3 + number4;

console.log(sum2);
```

Here, the DRY concept is completely disregarded, as the addition behavior is manually repeated.
While it may look innocent in such a trivial example, code duplication triggers significant problems as your application scales:

1. Maintenance Nightmares: Should you need to alter or fix the core logic, you are forced to track down and update every single duplicated occurrence manually.
2. Reduced Readability: Rampant code cloning clutters the application. When team members spot the same sequence of operations scattered across files, they might struggle to grasp its overarching intent, bringing confusion and increasing the likelihood of unforced errors.
3. Impeded Scalability: Repeating code undermines the very foundation of modular design. As your project evolves and business requirements shift, lacking a singular, reusable source of truth severely limits how efficiently your software can adapt and scale.

How can we refactor this effectively?

An optimized version adhering to the DRY principle:

```ts {1-3} title="example.ts"
const calculateSum = (a: number, b: number): number => {
  return a + b;
};

const sum1 = (1, 2);
const sum2 = (3, 4);

console.log(sum1);
console.log(sum2);
```

By crafting a reusable **calculateSum** function—which accepts two numeric arguments and outputs their total—we strictly adhere to the DRY standard.
This single function completely encapsulates the logic instead of scattering it throughout the file.
By doing so, we eradicate redundancy and boost reusability. Any future modifications to this addition process will only require adjusting one centralized location.

Implementing the DRY method drastically improves code quality and sustainable maintenance.
It accelerates development, minimizes the surface area for bugs, and keeps the workflow agile.
Committing to DRY practices empowers you to output pristine, modular code, yielding a more resilient architecture and a vastly superior developer experience.

### 3. You Aren't Gonna Need It (YAGNI)

YAGNI is a pragmatic guideline urging engineers to steer clear of building features that aren't strictly required right now.
It advocates for implementing solely the functionality dictated by current requirements, warning against speculative coding where developers anticipate arbitrary future needs that may never materialize.

Example:
Suppose you are tasked with creating a straightforward to-do list application.
You begin by establishing the fundamental abilities to simply create and remove tasks.
Midway through development, however, you feel compelled to integrate complex additions: priority tags, elaborate scheduling, push notifications, and multi-user collaboration.

By strictly adhering to YAGNI, you actively reject the urge to construct these unrequested features preemptively.
Instead, your priority remains shipping a functional Minimum Viable Product (MVP) that handles the baseline task management effectively.
You understand that shoehorning in speculative enhancements only balloons development timelines, inflates code complexity, and introduces needless bugs without delivering concrete, immediate value to the end user.

### 4. SOLID

The SOLID acronym represents five foundational design concepts intended to help developers structure software that is highly adaptable and easily maintained.
Each distinct principle targets a specific dimension of Object-Oriented design, collaboratively pushing engineers toward producing modular, scalable, and highly legible code architectures.

#### S - Single Responsibility Principle (SRP)

The SRP dictates that any given class or module must possess only one distinct responsibility—meaning it should have exactly one reason to undergo modifications.
This concept champions the idea that a class must serve a singular, focused objective, drastically simplifying its testing, comprehension, and upkeep.
By isolating behaviors and avoiding huge "god classes", we ensure pristine code readability and vastly superior long-term project health.

Example without following the SRP principle:

```kotlin title="Example.kt"
class TaskManager {
  fun manipulateTask(task: Task, action: String) {
    when (action) {
      "add" -> // Code to add task to the task manager
      "update" -> // Code to update task to the task manager
      "delete" -> // Code to delete task to the task manager
      else -> // Code to handle else condition
    }
  }
}

/**
 * The `TaskManager` class has multiple responsibilities, violating the SRP.
 * It is responsible for managing tasks, including adding, deleting, and updating them.
 * This violates the principle as the class has more than one reason to change.
 */
```

Example with following the SRP principle:

```kotlin title="Example.kt"
class TaskManager {
  fun addTask(task: Task) {
    // Code for adding task
  }

  fun updateTask(task: Task) {
    // Code for update task
  }

  fun deleteTask(task: Task) {
    // Code for delete task
  }
}

/**
 * The `TaskManager` class follows the SRP by having separate methods for adding, updating, and deleting tasks.
 * Each method has a single responsibility.
 */
```

#### O - Open/Closed Principle (OCP)

The OCP insists that software components—be it classes, modules, or functions—must be structured such that they are open for extension, yet strictly closed to direct modification.
Essentially, your architecture should facilitate the introduction of brand-new functionalities without requiring any alterations to the pre-existing logic.
By heavily utilizing polymorphism, interfaces, and abstract classes, you can seamlessly plug in new behaviors without risking regressions in stable, established code.

Example without following the OCP principle:

```kotlin title="Example.kt"
class PaymentProcessor {
  fun processPayment(paymentMethod: String) {
    when (paymentMethod) {
      "CreditCard" -> // Code for processing payment with credit card
      "Paypal" -> // Code for processing payment with Paypal
      else -> // Do something on else condition
    }
  }
}

/**
 * The `PaymentProcessor` class violates the OCP by being open for modification.
 * If a new payment method is introduced, the class would need to be modified to handle the new logic, breaking the principle.
 */
```

Example with following the OCP principle:

```kotlin title="Example.kt"
interface PaymentProcessor {
  fun processPayment()
}

class CreditCardProcessor: PaymentProcessor {
  override fun processPayment() {
    // Code for processing payment with credit card
  }
}

class PaypalProcessor: PaymentProcessor {
  override fun processPayment() {
    // Code for processing payment with Paypal
  }
}

/**
 * The `PaymentProcessor` interface and its implementations (CreditCardProcessor and PaypalProcessor) adhere to the OCP.
 * If we want to add a new payment processor, we can simply create a new class implementing the `PaymentProcessor`
 * interface without modifying the existing code.
 */
```

#### L - Liskov Substitution Principle (LSP)

The LSP decrees that any child class or subtype must be entirely substitutable for its parent/base class without compromising the program's accuracy or stability.
Put simply, a derived class should be capable of effortlessly stepping into the shoes of its ancestor class without violating expected behaviors or triggering runtime failures.
Following the LSP tightly guarantees safe object interchangeability and drastically improves the reliability of reusable components.

Example without following the LSP principle:

```kotlin title="Example.kt"
class Vehicle {
  fun startEngine() {
    // Code to start the engine
  }
}

class Car: Vehicle {
  override fun startEngine() {
    // Code specific to starting a car's engine
  }
}

class Motorcycle: Vehicle {
  override fun startEngine() {
    // Code specific to starting a motorcycle's engine
  }
}

/**
 * The `Car` and `Motorcycle` classes violate the LSP by adding specific behavior to their respective startEngine methods.
 * This could lead to unexpected behavior when substituting objects of the base class with objects of the derived classes.
 */
```

Example with following the LSP principle:

```kotlin title="Example.kt"
abstract class Vehicle {
  abstract fun startEngine() {
    // Code to start the engine
  }
}

class Car: Vehicle {
  override fun startEngine() {
    // Code specific to starting a car's engine
  }
}

class Motorcycle: Vehicle {
  override fun startEngine() {
    // Code specific to starting a motorcycle's engine
  }
}

/**
 * The `Car` and `Motorcycle` classes are substitutable for their base class `Vehicle` without
 * changing the behavior of the program.
 * They both override the `startEngine` method based on their specific implementation.
 */
```

#### I - Interface Segregation Principle (ISP)

The ISP mandates that no client should ever be compelled to rely on methods or interfaces that it does not actively utilize.
It strongly recommends the creation of highly specialized, granular interfaces as opposed to a single, monolithic, "catch-all" interface.
By intelligently splintering interfaces to serve exact client requirements, we eliminate bloated dependencies, unlocking deeper modularity and dynamic flexibility.

Example without following the ISP principle:

```kotlin title="Example.kt"
interface PrintAndScan {
  fun print()
  fun scan()
}

class AllInOnePrinter: PrintAndScan {
  override fun print() {
    // Logic for printing
  }

  override fun scan() {
    // Login for scan
  }
}

/**
 * The `PrintAndScan` interface violates the ISP as clients might be forced to
 * implement methods they don't need, introducing unnecessary dependencies.
 */
```

Example with following the ISP principle:

```kotlin title="Example.kt"
interface Printer {
  fun print()
}

interface Scanner {
  fun scan()
}

class AllInOnePrinter: Printer, Scanner {
  override fun print() {
    // Logic for printing
  }

  override fun scan() {
    // Logic for scanning
  }
}

/**
 * The code follows the ISP by segregating the interfaces `Printer` and `Scanner` based on specific client needs.
 * The `AllInOnePrinter` class implements both interfaces, providing the necessary functionality.
 */
```

#### D - Dependency Inversion Principle (DIP)

The DIP declares that high-level architectural modules must never rely directly on low-level implementation modules; rather, both tiers must depend entirely on abstract definitions.
It champions the usage of dependency injection and abstract interfaces to sever tight restraints, fostering loose coupling across the entire system.
By pivoting dependencies toward abstractions, we empower ourselves to swiftly swap out underlying implementations, rendering the software far more agile and test-friendly.

Example without following the DIP principle:

```kotlin title="Example.kt"
class EmailService {
  fun sendEmail(message: String) {
    // Code to send email
  }
}

class NotificationService {
  private val emailService = EmailService()

  fun sendNotification(message: String) {
    emailService.sendEmail(message)
  }
}

/**
 * The `NotificationService` directly depends on the `EmailService` class, violating the DIP.
 * This tight coupling makes it harder to switch to different notification mechanisms without modifying the service.
 */
```

Example with following the DIP principle:

```kotlin title="Example.kt"
interface Notifier {
  fun notify(message: String)
}

class EmailNotifier: Notifier {
  override fun notify(message: String) {
    // Code to send an email notification
  }
}

class SMSNotifier: Notifier {
  override fun notify(message: String) {
    // Code to send an SMS notification
  }
}

class NotificationService(private val notifier: Notifier) {
  fun sendNotification(message: String) {
    notifier.notify(message)
  }
}

/**
 * The `NotificationService` depends on the `Notifier` interface,
 * allowing different implementations (`EmailNotifier`, `SMSNotifier`) to be injected.
 * This promotes loose coupling between the service and the concrete implementations.
 */
```

By deeply comprehending and actively operating under the SOLID tenets, developers can systematically construct highly modular, resilient, and adaptive frameworks.
Adhering to these structural dogmas naturally yields systems that are a breeze to test, decipher, and scale, ultimately forging exceptionally robust enterprise software.

### 5. Separation of Concerns (SOC)

The Separation of Concerns (SoC) principle stands as a bedrock concept advocating for the segmentation of a large software application into distinct, sovereign boundaries—where each slice tackles an isolated functional concern.
The core objective is to untangle overlapping responsibilities to cultivate strict modularity, ease of maintenance, and high reusability.

Example:
Consider the construction of a house. The property is comprised of multiple independent systems: foundational concrete, structural walls, roofing, plumbing networks, electrical grids, and aesthetic interiors.
Every single element serves its own unambiguous purpose, yet they all harmonize to form a complete, livable residence.

Implementing SoC in software engineering mirrors this architectural marvel.
Instead of weaving a monolithic, highly interdependent web of code, we methodically fracture the application into autonomous modules, classes, and helper functions—each meticulously focused on resolving one specific domain problem.
